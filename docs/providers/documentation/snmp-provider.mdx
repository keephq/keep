## Overview

The SNMP provider ingests **forwarded JSON trap payloads** into Keep. In this version, Keep does not listen for native UDP traps directly. Instead, you run an external trap receiver such as `snmptrapd`, transform the trap into JSON, and `POST` it to Keep at `/alerts/event/snmp`.

## Forwarded Payload Format

Keep accepts a single JSON object or a JSON array of trap objects. These fields are recognized:

- `trap_oid` or `oid`
- `source_address`
- `timestamp`
- `trap_name`
- `message`
- `description`
- `entity_id`
- `variables` or `varbinds`
- `fingerprint`

Example payload:

```json
{
  "trap_oid": "1.3.6.1.6.3.1.1.5.3",
  "source_address": "192.0.2.10",
  "timestamp": "2026-03-25T10:00:00Z",
  "variables": {
    "ifIndex": "12",
    "ifName": "xe-0/0/0",
    "ifDescr": "uplink to core"
  }
}
```

## Example `snmptrapd` Setup

One common pattern is to let `snmptrapd` receive native UDP traps and forward a structured JSON payload into Keep.

Example `snmptrapd.conf`:

```conf
disableAuthorization yes
authCommunity log,execute public
traphandle default /usr/local/bin/forward-snmp-to-keep.sh
```

Your forwarder script is responsible for translating the raw trap into the JSON shape above and posting it to Keep with an `x-api-key` header. For interface-level notifications, include `entity_id` or at least `ifIndex`, `ifName`, or `ifDescr` in `variables` so Keep can build a stable fingerprint.

## Behavior Notes

- `linkDown` becomes a `critical` firing alert.
- `linkUp` becomes an `info` firing alert.
- Unknown trap OIDs default to `warning`.
- This version does **not** perform automatic resolution or native SNMP trap listening inside Keep.
