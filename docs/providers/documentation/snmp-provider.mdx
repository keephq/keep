## Overview

The SNMP provider receives decoded SNMP trap events as JSON and converts them into Keep alerts.

Keep does not open a UDP listener for this provider. SNMP traps are normally emitted over UDP, often on privileged port 162, so the recommended deployment is to run a trap receiver such as `snmptrapd`, `snmptt`, Telegraf, or another bridge near the devices and forward decoded trap JSON to Keep's webhook endpoint.

## Webhook Endpoint

Send decoded traps to:

```bash
https://api.keephq.dev/alerts/event/snmp
```

Include a Keep API key with webhook permissions:

```bash
X-API-KEY: <keep-webhook-api-key>
```

## Example Request

```bash
curl -X POST "$KEEP_API_URL/alerts/event/snmp" \
  -H "Content-Type: application/json" \
  -H "X-API-KEY: $KEEP_WEBHOOK_API_KEY" \
  -d '{
    "version": "2c",
    "source_ip": "10.0.0.15",
    "agent_address": "10.0.0.15",
    "community": "public",
    "enterprise_oid": "1.3.6.1.4.1.8072.3.2.10",
    "trap_oid": "1.3.6.1.6.3.1.1.5.3",
    "generic_trap": "linkDown",
    "specific_trap": 0,
    "timestamp": "2026-05-14T12:00:00Z",
    "varbinds": [
      {
        "oid": "1.3.6.1.2.1.2.2.1.2.1",
        "name": "ifDescr",
        "value": "eth0"
      }
    ]
  }'
```

This creates a firing critical alert named `SNMP linkDown on 10.0.0.15`. The trap OID and varbinds are available in alert labels and structured fields.

## Payload Fields

The provider accepts this canonical shape:

```json
{
  "version": "2c",
  "source_ip": "10.0.0.15",
  "agent_address": "10.0.0.15",
  "community": "public",
  "enterprise_oid": "1.3.6.1.4.1.8072.3.2.10",
  "trap_oid": "1.3.6.1.6.3.1.1.5.3",
  "generic_trap": "linkDown",
  "specific_trap": 0,
  "timestamp": "2026-05-14T12:00:00Z",
  "varbinds": [
    {
      "oid": "1.3.6.1.2.1.2.2.1.2.1",
      "name": "ifDescr",
      "value": "eth0"
    }
  ]
}
```

Common aliases are also accepted:

| Value | Accepted fields |
|:------|:----------------|
| Host or source | `host`, `hostname`, `source_ip`, `agent_address`, `source`, `agent`, `ip` |
| Trap OID | `trap_oid`, `snmpTrapOID`, `snmp_trap_oid`, `oid`, `trapOid` |
| Generic trap | `generic_trap`, `trap_type`, `genericTrap`, `generic_trap_type` |
| Timestamp | `timestamp`, `time`, `received_at`, `lastReceived`, `last_received` |
| Varbinds | `varbinds`, `vars`, `variable_bindings`, `bindings` |

Varbinds can be sent as objects, two-item arrays, or an object keyed by OID:

```json
{
  "varbinds": [
    {"oid": "1.3.6.1.2.1.2.2.1.2.1", "name": "ifDescr", "value": "eth0"},
    ["1.3.6.1.2.1.2.2.1.8.1", "down"]
  ],
  "variable_bindings": {
    "1.3.6.1.2.1.2.2.1.2.1": "eth0"
  }
}
```

## Trap Mapping

| SNMP trap | Keep status | Keep severity |
|:----------|:------------|:--------------|
| `coldStart` | `firing` | `info` |
| `warmStart` | `firing` | `info` |
| `linkDown` | `firing` | `critical` |
| `linkUp` | `resolved` | `info` |
| `authenticationFailure` | `firing` | `warning` |
| `egpNeighborLoss` | `firing` | `critical` |
| `enterpriseSpecific` | `firing` | payload `severity` or `info` |
| unknown | `firing` | payload `severity` or `info` |

Explicit `status` and `severity` fields override the default mapping after normalization. For example, `ok` maps to `resolved`, and `crit` maps to `critical`.

## Forwarding From snmptrapd

One common pattern is to let `snmptrapd` receive UDP traps and invoke a local script that posts JSON to Keep.

Example `snmptrapd.conf` entry:

```text
disableAuthorization yes
traphandle default /usr/local/bin/keep-snmp-forwarder.sh
```

Example forwarder outline:

```bash
#!/usr/bin/env bash
set -euo pipefail

raw_trap="$(cat)"

curl -sS -X POST "$KEEP_API_URL/alerts/event/snmp" \
  -H "Content-Type: application/json" \
  -H "X-API-KEY: $KEEP_WEBHOOK_API_KEY" \
  -d "$(jq -n --arg raw "$raw_trap" '{
    source: "snmptrapd",
    generic_trap: "enterpriseSpecific",
    message: $raw,
    varbinds: [
      {"name": "raw_trap", "value": $raw}
    ]
  }')"
```

Production forwarders should parse the trap OID, source address, and varbinds from the trap receiver output and populate the canonical fields shown above.

## Alert Details

The provider sets:

- `source` to `["snmp"]`
- `host`, `hostname`, and `service` from the normalized source host
- `trap_oid`, `generic_trap`, `specific_trap`, and `enterprise_oid` as structured fields
- `snmp_varbinds` with normalized varbind objects
- `snmp_event` with the raw forwarded payload
- labels for host, trap metadata, named varbinds, and OID-based varbind keys

Alert IDs and fingerprints are stable when an explicit ID is not provided. They are derived from the source host, trap OID, generic trap, specific trap, and normalized varbinds.
