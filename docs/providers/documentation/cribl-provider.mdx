## Overview

[Cribl](https://cribl.io/) is an observability pipeline platform (Cribl Stream, Cribl Edge, Cribl Search) that collects, transforms, and routes logs, metrics, and traces from any source to any destination. Cribl sits between your data sources and observability tools, giving you full control over what data gets forwarded, enriched, or filtered.

The Keep Cribl provider receives processed events from Cribl Stream or Cribl Edge via an **HTTP output destination**, allowing you to:

- Forward alerts and critical events from any source (Splunk, CloudWatch, syslog, Kafka, etc.) through Cribl into Keep
- Apply Cribl pipeline transforms and enrichment before events reach Keep
- Receive Cribl platform notifications (backpressure alerts, worker health, search completions)

## Connecting with the Provider

The Cribl provider is **webhook-only** — Keep listens for incoming POST requests from Cribl's HTTP output destination.

### Configuration

| Field | Required | Description |
|-------|----------|-------------|
| `webhook_api_key` | No | Optional API key. Set this to match the `Authorization` header token you configure in the Cribl HTTP destination. |

## Webhook Setup

### Cribl Stream / Cribl Edge — HTTP Output

1. In Cribl, go to **Routing → QuickConnect** or open a pipeline in **Processing → Pipelines**.
2. Click **+ Add Destination → HTTP**.
3. Configure the destination:

   | Setting | Value |
   |---------|-------|
   | **URL** | `https://api.keephq.dev/alerts/event/cribl` |
   | **Method** | `POST` |
   | **Content-Type header** | `application/json` |
   | **x-api-key header** | `<your-keep-api-key>` |
   | **Format** | JSON |

4. Save and deploy.

### Recommended event enrichment

Before routing events to Keep, add an **Eval** function in your Cribl pipeline to ensure events contain:

```javascript
// Cribl pipeline Eval function
__e['severity'] = __e['severity'] || __e['level'] || 'info';
__e['status'] = __e['status'] || 'firing';
__e['service'] = __e['service'] || __e['sourcetype'] || __e['source'];
```

### Cribl.Cloud Notifications

Cribl.Cloud platform notifications (backpressure, worker health, etc.) can be forwarded directly to Keep. In **Cribl.Cloud → Settings → Notifications**, add a webhook destination pointing to:

```
https://api.keephq.dev/alerts/event/cribl
```

## Event Payload

The Cribl provider accepts flexible JSON payloads. Fields are extracted from a range of common field names to maximise compatibility with upstream source formats.

### Generic alert event

```json
{
  "_time": 1705312200.0,
  "host": "prod-worker-01",
  "source": "app-monitor",
  "name": "HighErrorRate",
  "message": "Error rate exceeded 15% in the last 5 minutes",
  "severity": "critical",
  "status": "firing",
  "service": "checkout-service",
  "environment": "production"
}
```

### Splunk HEC passthrough

```json
{
  "_time": 1705312800.0,
  "host": "web-server-02",
  "sourcetype": "access_combined",
  "source": "/var/log/nginx/access.log",
  "severity": "warning",
  "status": "firing",
  "message": "HTTP 5xx error rate spiking: 203 errors in 60s"
}
```

### Cribl platform notification

```json
{
  "type": "backpressure",
  "condition": "output_queue_full",
  "value": "98%",
  "description": "HTTP output queue is 98% full — events may be dropped",
  "_time": 1705313400.0,
  "host": "cribl-leader-01"
}
```

## Field Mapping

| Cribl field(s) | Keep AlertDto field |
|----------------|---------------------|
| `id`, `_id`, `event_id` | `id` |
| `name`, `alert_name`, `sourcetype`, `source`, `logGroup` | `name` |

| `service`, `source`, `sourcetype` | `service` |

| `status`, `state` | `status` |
