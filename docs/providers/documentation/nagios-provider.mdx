## Overview

[Nagios](https://www.nagios.org/) is a widely-used open-source infrastructure and network monitoring system that has been the industry standard for over two decades. Nagios monitors hosts and services, alerting teams when things go wrong and when they recover.

The Keep Nagios provider receives host and service alert notifications from Nagios Core or Nagios XI via a lightweight HTTP webhook notification command.

**Supported features:**
- Receive host check alerts (UP / DOWN / UNREACHABLE)
- Receive service check alerts (OK / WARNING / CRITICAL / UNKNOWN)
- Detect recoveries and acknowledgements via `notification_type`
- Works with Nagios Core 3/4 and Nagios XI

## Connecting with the Provider

The Nagios provider is webhook-only — Keep listens for incoming POST requests from Nagios. No API key is required for the provider itself, though you can optionally configure one to authenticate incoming requests.

### Configuration

| Field | Required | Description |
|-------|----------|-------------|
| `webhook_api_key` | No | Optional API key sent in the `x-api-key` header from Nagios to authenticate webhook requests |

## Webhook Setup

Nagios sends notifications via shell commands. The following steps configure Nagios to POST alert data to Keep.

### Step 1 — Create the notification script

Save the following script to your Nagios server, e.g. `/usr/local/nagios/libexec/notify_keep.sh`:

```bash
#!/bin/bash
TYPE="$1"          # HOST or SERVICE
HOST="$2"
HOST_ADDR="$3"
STATE="$4"
SERVICE="$5"
OUTPUT="$6"
NOTIF_TYPE="$7"
ATTEMPT="$8"
MAX_ATTEMPTS="$9"

curl -s -X POST "https://api.keephq.dev/alerts/event/nagios" \
  -H "Content-Type: application/json" \
  -H "x-api-key: <your-keep-api-key>" \
  -d "{
    \"check_type\": \"$TYPE\",
    \"hostname\": \"$HOST\",
    \"host_address\": \"$HOST_ADDR\",
    \"state\": \"$STATE\",
    \"service_description\": \"$SERVICE\",
    \"plugin_output\": \"$OUTPUT\",
    \"notification_type\": \"$NOTIF_TYPE\",
    \"check_attempt\": $ATTEMPT,
    \"max_check_attempts\": $MAX_ATTEMPTS
  }"
```

Make it executable:

```bash
chmod +x /usr/local/nagios/libexec/notify_keep.sh
```

### Step 2 — Register the notification commands

Add the following to your `commands.cfg`:

```cfg
define command {
  command_name  notify-keep-host
  command_line  /usr/local/nagios/libexec/notify_keep.sh HOST $HOSTNAME$ $HOSTADDRESS$ $HOSTSTATE$ "" "$HOSTOUTPUT$" $NOTIFICATIONTYPE$ $HOSTSTATEID$ $MAXHOSTCHECKS$
}

define command {
  command_name  notify-keep-service
  command_line  /usr/local/nagios/libexec/notify_keep.sh SERVICE $HOSTNAME$ $HOSTADDRESS$ $SERVICESTATE$ $SERVICEDESC$ "$SERVICEOUTPUT$" $NOTIFICATIONTYPE$ $SERVICEATTEMPT$ $MAXSERVICECHECKS$
}
```

### Step 3 — Create a Keep contact

Add a contact to your `contacts.cfg`:

```cfg
define contact {
  contact_name                    keep-notify
  alias                           Keep Alerting
  host_notification_commands      notify-keep-host
  service_notification_commands   notify-keep-service
  host_notification_period        24x7
  service_notification_period     24x7
  host_notification_options       d,u,r,f
  service_notification_options    w,u,c,r,f
  email                           (none)
}
```

### Step 4 — Assign the contact to hosts or contact groups

Add `keep-notify` to the `contacts` or `contact_groups` directive on any host or service definition you want forwarded to Keep:

```cfg
define host {
  host_name   web-server-01
  contacts    keep-notify,your-other-contacts
  ...
}
```

Reload Nagios:

```bash
systemctl reload nagios
# or: /usr/local/nagios/bin/nagios -v /usr/local/nagios/etc/nagios.cfg && systemctl reload nagios
```

## Alert Payload

Example host DOWN payload that Nagios sends to Keep:

```json
{
  "check_type": "HOST",
  "hostname": "web-server-01",
  "host_address": "192.168.1.10",
  "state": "DOWN",
  "notification_type": "PROBLEM",
  "plugin_output": "PING CRITICAL - Packet loss = 100%",
  "check_attempt": 3,
  "max_check_attempts": 3
}
```

Example service CRITICAL payload:

```json
{
  "check_type": "SERVICE",
  "hostname": "db-server-02",
  "host_address": "192.168.1.20",
  "state": "CRITICAL",
  "service_description": "MySQL",
  "notification_type": "PROBLEM",
  "plugin_output": "CRITICAL: MySQL is not running",
  "check_attempt": 3,
  "max_check_attempts": 3
}
```

## State and Severity Mapping

### Host checks

| Nagios state | Keep severity | Keep status |
|-------------|---------------|-------------|
| `UP` | Info | Resolved |
| `DOWN` | Critical | Firing |
| `UNREACHABLE` | High | Firing |

### Service checks

| Nagios state | Keep severity | Keep status |
|-------------|---------------|-------------|
| `OK` | Info | Resolved |
| `WARNING` | Warning | Firing |
| `CRITICAL` | Critical | Firing |
| `UNKNOWN` | High | Firing |

### Notification type overrides

| `notification_type` | Keep status |
|--------------------|-------------|
| `PROBLEM` | Firing |
| `RECOVERY` | Resolved |
| `ACKNOWLEDGEMENT` | Acknowledged |
| `FLAPPINGSTART` | Firing |
| `FLAPPINGSTOP` | Resolved |
