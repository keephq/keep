# Nagios Provider

The Nagios provider allows you to receive alerts from [Nagios XI](https://www.nagios.com/products/nagios-xi/) and [Nagios Core](https://www.nagios.org/) monitoring systems within Keep.

The provider supports two methods for receiving alerts:
- **API Polling** (Nagios XI only) — Pull host and service problems via the REST API
- **Webhooks** (Nagios XI and Core) — Push notifications in real time via curl

## Setup

### Prerequisites
1. A Nagios XI instance (for API polling) or Nagios Core/XI (for webhooks)
2. A Nagios XI API key (Admin → Manage API Keys)
3. A Keep instance with webhook capability (for webhook mode)

### Configuration (API Polling)

```yaml
authentication:
  host_url: "https://nagios.example.com"  # Your Nagios XI base URL
  api_key: "your-nagios-xi-api-key"       # Admin → Manage API Keys
```

The provider polls the `/nagiosxi/api/v1/objects/hoststatus` and
`/nagiosxi/api/v1/objects/servicestatus` endpoints to fetch current problems.

### Webhook Configuration

To send real-time alerts from Nagios to Keep via webhooks, create a
notification command that POSTs JSON to your Keep webhook URL.

**Nagios XI / Core notification command:**

```bash
define command {
    command_name    notify-keep-service
    command_line    /usr/bin/curl -s -X POST \
        -H "Content-Type: application/json" \
        -H "X-API-KEY: $USER10$" \
        -d '{ \
            "host_name": "$HOSTNAME$", \
            "host_alias": "$HOSTALIAS$", \
            "host_address": "$HOSTADDRESS$", \
            "host_state": "$HOSTSTATE$", \
            "host_state_id": "$HOSTSTATEID$", \
            "service_description": "$SERVICEDESC$", \
            "service_state": "$SERVICESTATE$", \
            "service_state_id": "$SERVICESTATEID$", \
            "service_output": "$SERVICEOUTPUT$", \
            "long_service_output": "$LONGSERVICEOUTPUT$", \
            "notification_type": "$NOTIFICATIONTYPE$", \
            "timestamp": "$TIMET$", \
            "attempt": "$SERVICEATTEMPT$", \
            "max_attempts": "$MAXSERVICEATTEMPTS$", \
            "downtime_depth": "$SERVICEDOWNTIMEDEPTH$" \
        }' \
        "https://your-keep-instance/alerts/event/nagios"
}
```

**Host notification command:**

```bash
define command {
    command_name    notify-keep-host
    command_line    /usr/bin/curl -s -X POST \
        -H "Content-Type: application/json" \
        -H "X-API-KEY: $USER10$" \
        -d '{ \
            "host_name": "$HOSTNAME$", \
            "host_alias": "$HOSTALIAS$", \
            "host_address": "$HOSTADDRESS$", \
            "host_state": "$HOSTSTATE$", \
            "host_state_id": "$HOSTSTATEID$", \
            "notification_type": "$NOTIFICATIONTYPE$", \
            "host_output": "$HOSTOUTPUT$", \
            "long_host_output": "$LONGHOSTOUTPUT$", \
            "timestamp": "$TIMET$" \
        }' \
        "https://your-keep-instance/alerts/event/nagios"
}
```

Then assign these commands to your contact notification settings:

```bash
define contact {
    contact_name                    keep
    service_notification_commands   notify-keep-service
    host_notification_commands      notify-keep-host
    # ... other contact settings
}
```

### Nagios XI vs Nagios Core

| Feature | Nagios XI | Nagios Core |
|:--------|:----------|:------------|
| API Polling | ✅ REST API | ❌ No API |
| Webhooks | ✅ | ✅ |
| Auth Method | API Key | N/A (webhook only) |

### State Mapping

Nagios states are automatically mapped to Keep alert statuses and severities:

<TableWrapper>
#### Status Mapping
| Nagios State | Keep Status |
|:-------------|:------------|
| UP           | RESOLVED    |
| DOWN         | FIRING      |
| UNREACHABLE  | FIRING      |
| OK           | RESOLVED    |
| WARNING      | FIRING      |
| CRITICAL     | FIRING      |
| UNKNOWN      | FIRING      |

</TableWrapper>
