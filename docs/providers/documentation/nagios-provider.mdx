# Nagios Provider

The Nagios provider allows you to receive alerts from Nagios monitoring system within Keep.
Nagios provider supports 2 methods for receiving alerts: Webhooks & API Polling.

The recommended and primary method for receiving alerts is via Webhooks.

## Setup

### Prerequisites
1. Access to a Nagios instance (Nagios Core or Nagios XI)
2. Keep instance with webhook capability

### Configuration

For webhook-only usage, no authentication configuration is required.

For API polling (optional), the provider requires:

```yaml
authentication:
  host_url: "https://nagios.example.com"  # Your Nagios instance URL
  api_user: "your-api-user"               # Nagios API username (optional)
  api_password: "your-api-key"            # Nagios XI API key (optional)
```

### Webhook Configuration

To configure Nagios to send alerts to Keep via webhooks:

1. Create a notification command in your Nagios configuration

For service notifications:
```bash
#!/bin/bash
# /usr/local/nagios/libexec/notify-service-by-keep.sh

curl -X POST "{keep_webhook_url}" \
  -H "Content-Type: application/json" \
  -H "X-API-KEY: {keep_api_key}" \
  -d "{
    \"notification_type\": \"$NOTIFICATIONTYPE$\",
    \"host_name\": \"$HOSTNAME$\",
    \"host_address\": \"$HOSTADDRESS$\",
    \"host_state\": \"$HOSTSTATE$\",
    \"host_output\": \"$HOSTOUTPUT$\",
    \"service_description\": \"$SERVICEDESC$\",
    \"service_state\": \"$SERVICESTATE$\",
    \"service_output\": \"$SERVICEOUTPUT$\",
    \"last_check\": \"$LASTSERVICECHECK$\",
    \"long_date_time\": \"$LONGDATETIME$\"
  }"
```

For host notifications:
```bash
#!/bin/bash
# /usr/local/nagios/libexec/notify-host-by-keep.sh

curl -X POST "{keep_webhook_url}" \
  -H "Content-Type: application/json" \
  -H "X-API-KEY: {keep_api_key}" \
  -d "{
    \"notification_type\": \"$NOTIFICATIONTYPE$\",
    \"host_name\": \"$HOSTNAME$\",
    \"host_address\": \"$HOSTADDRESS$\",
    \"host_state\": \"$HOSTSTATE$\",
    \"host_output\": \"$HOSTOUTPUT$\",
    \"last_check\": \"$LASTHOSTCHECK$\",
    \"long_date_time\": \"$LONGDATETIME$\"
  }"
```

2. Define the commands in your Nagios configuration:
```cfg
define command {
    command_name    notify-service-by-keep
    command_line    /usr/local/nagios/libexec/notify-service-by-keep.sh
}

define command {
    command_name    notify-host-by-keep
    command_line    /usr/local/nagios/libexec/notify-host-by-keep.sh
}
```

3. Assign the commands to your contact or contact group:
```cfg
define contact {
    contact_name                    keep-contact
    service_notification_commands   notify-service-by-keep
    host_notification_commands      notify-host-by-keep
    ...
}
```

4. Reload Nagios to apply changes.

### API Polling (Optional)

For Nagios XI users, Keep can optionally pull service status via the Nagios XI API:

1. Obtain an API key from Nagios XI (Admin → System Extensions → API Keys)
2. Configure `host_url` and `api_password` (API key) in Keep provider settings
3. Enable pulling in the Keep provider configuration

### State Mapping

Nagios states are automatically mapped to Keep alert severities and statuses:

#### Service Status Mapping
| Nagios State  | Keep Status |
|:-------------|:------------|
| OK           | RESOLVED    |
| WARNING      | FIRING      |
| CRITICAL     | FIRING      |
| UNKNOWN      | FIRING      |

#### Host Status Mapping
| Nagios State    | Keep Status |
|:---------------|:------------|
| UP             | RESOLVED    |
| DOWN           | FIRING      |
| UNREACHABLE    | FIRING      |

#### Severity Mapping
| Nagios State    | Keep Severity |
|:---------------|:--------------|
| OK             | INFO          |
| WARNING        | WARNING       |
| CRITICAL       | CRITICAL      |
| UNKNOWN        | INFO          |
| UP             | INFO          |
| DOWN           | CRITICAL      |
| UNREACHABLE    | WARNING       |
