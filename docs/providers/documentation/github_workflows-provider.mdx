## Connecting GitHub Workflows to Keep

The GitHub Workflows provider monitors your GitHub Actions workflow runs and creates alerts when workflows fail, time out, are cancelled, or require action.

### Prerequisites

1. A GitHub Personal Access Token (PAT) with appropriate permissions
2. Repository access to the workflows you want to monitor

### Setup Steps

#### 1. Create a GitHub Personal Access Token

1. Go to GitHub Settings → Developer settings → Personal access tokens → Tokens (classic)
2. Click "Generate new token (classic)"
3. Give your token a descriptive name like "Keep GitHub Workflows Monitor"
4. Select the following scopes:
   - `repo` (Full control of private repositories) - OR - `public_repo` (Access public repositories) if monitoring only public repos
   - `actions:read` (Read access to actions and workflows)

  <img height="10" src="/images/github-pat-creation.png" />
</Frame>

5. Click "Generate token" and copy the generated token immediately (you won't be able to see it again)

#### 2. Configure the Provider in Keep

1. Go to Keep dashboard
2. Navigate to "Providers" and click "Add Provider"
3. Select "GitHub Workflows" from the list
4. Fill in the required fields:
   - **Access Token**: Paste your GitHub Personal Access Token
   - **Repositories**: Comma-separated list of repositories to monitor (e.g., `owner/repo1,owner/repo2`)
   - **GitHub Base URL** (optional): Only needed for GitHub Enterprise Server (defaults to `https://api.github.com`)

<Frame width="100" height="200">
  <img height="10" src="/images/github-workflows-provider-config.png" />
</Frame>

5. Click "Connect" to validate and save the configuration

#### 3. Repository Format

When specifying repositories, use the format `owner/repository-name`. Examples:

- Single repository: `microsoft/vscode`
- Multiple repositories: `facebook/react,microsoft/vscode,google/tensorflow`
- Organization repositories: `myorg/frontend,myorg/backend,myorg/api`

### Supported GitHub Enterprise Server

This provider supports GitHub Enterprise Server installations. Simply provide your enterprise GitHub API URL in the "GitHub Base URL" field:

```
https://github.your-company.com/api/v3
```

### Alert Mapping

The provider maps GitHub Actions workflow conclusions to Keep alert severities and statuses:

#### Severity Mapping
- **failure** → CRITICAL
- **timed_out** → HIGH  
- **cancelled** → LOW
- **action_required** → WARNING

#### Status Mapping
- **failure/timed_out** → FIRING
- **success** → RESOLVED
- **cancelled/action_required** → FIRING

### Alert Details

Each alert includes the following information:

- **Name**: Workflow name
- **Description**: "{workflow_name} failed in {repository}"
- **Repository**: The repository where the workflow ran
- **Workflow ID**: GitHub's internal workflow ID
- **Run Number**: The sequential run number
- **Conclusion**: The workflow's conclusion (failure, timed_out, etc.)
- **Branch**: The branch where the workflow ran
- **Actor**: The user who triggered the workflow
- **Event**: The event that triggered the workflow (push, pull_request, etc.)
- **URL**: Direct link to the workflow run on GitHub

### Deduplication

The provider uses fingerprinting to prevent duplicate alerts:
- **Fingerprint Format**: `{repository}:{workflow_id}:{run_number}`

This ensures that each unique workflow run creates only one alert, even if the provider polls multiple times.

### Troubleshooting

#### Authentication Issues

If you receive authentication errors:

1. Verify your Personal Access Token is correct and hasn't expired
2. Ensure the token has the required scopes (`actions:read` and repository access)
3. For private repositories, ensure the `repo` scope is enabled
4. For GitHub Enterprise Server, verify the base URL is correct

#### No Alerts Appearing

1. Check that the repository names are correctly formatted (`owner/repo`)
2. Verify the repositories have GitHub Actions workflows
3. Ensure there are recent workflow runs with failure conclusions
4. Check the Keep logs for any error messages from the provider

#### Rate Limiting

GitHub API has rate limits:
- 5,000 requests per hour for authenticated requests
- The provider fetches up to 100 workflow runs per repository per poll
- Consider the number of repositories and polling frequency to stay within limits

### Best Practices

1. **Monitor Critical Repositories**: Focus on repositories with important workflows (deployments, releases, etc.)
2. **Use Descriptive Token Names**: Name your PAT clearly for easier management
3. **Regular Token Rotation**: Rotate your access tokens periodically for security
4. **Repository Grouping**: Group related repositories to manage alerts effectively
5. **Filtering**: Use Keep's alert filtering and routing to send different workflow alerts to appropriate channels
