## Overview

[OpenSupports](https://www.opensupports.com/) is an open-source, self-hosted ticketing and customer support system. It provides a REST API for programmatic ticket management, making it a great ITSM tool for teams that want to track incidents as support tickets.

The Keep OpenSupports provider supports:
- **Action (notify)**: Create a ticket in OpenSupports when a Keep alert fires or a workflow step runs
- **Pull**: Fetch open tickets from OpenSupports and display them in Keep as alert-like records

This lets Keep serve as a bridge between your observability stack (Prometheus, Grafana, Datadog, etc.) and your IT service management system — every critical alert automatically becomes a tracked ticket.

## Connecting with the Provider

### Prerequisites

1. A running OpenSupports instance (self-hosted)
2. A staff account with API access enabled

### Generate an API Key

1. Log into OpenSupports as an administrator
2. Go to **Staff** → click on the staff member you want to use
3. Click **Edit** → find the **API Key** field
4. Generate or copy the API key

### Configuration

| Field | Required | Description |
|-------|----------|-------------|
| `host` | Yes | OpenSupports base URL (e.g., `https://support.example.com`) |
| `api_key` | Yes | Staff API key generated in the OpenSupports staff profile |
| `staff_email` | Yes | Email address of the staff account that owns the API key |
| `department_id` | No | Default department ID for new tickets (default: `1`) |
| `verify_ssl` | No | Verify SSL certificates (default: `true`) |

### Finding Department IDs

Department IDs can be found in OpenSupports under **Admin → Departments**. The ID is shown in the URL when you click on a department.

## Usage in Workflows

### Create a ticket when an alert fires

```yaml
workflow:
  id: opensupports-ticket-on-alert
  name: Create OpenSupports ticket for critical alerts
  triggers:
    - type: alert
      filters:
        - key: severity
          value: critical
  steps:
    - name: create-ticket
      provider:
        type: opensupports
        config: "{{ providers.opensupports }}"
      with:
        title: "ALERT: {{ alert.name }} — {{ alert.severity }}"
        description: |
          Keep detected a critical alert that requires attention.

          **Alert:** {{ alert.name }}
          **Severity:** {{ alert.severity }}
          **Source:** {{ alert.source }}
          **Description:** {{ alert.description }}
          **Time:** {{ alert.lastReceived }}
        priority: 2
        department_id: "1"
        author_name: "Keep Automation"
        author_email: "keep@example.com"
```

### Ticket priority values

| OpenSupports priority | Keep severity equivalent | Value |
|-----------------------|--------------------------|-------|
| High | Critical / High | `2` |
| Medium | Warning | `1` |
| Low | Info / Low | `0` |

## Pulling Tickets

When Keep pulls tickets from OpenSupports, open tickets are mapped to Keep alert records:

| OpenSupports field | Keep AlertDto field |
|-------------------|---------------------|
| `ticketNumber` | `id`, `fingerprint` |
| `title` | `name` |
| `content` | `description` |
| `priority` (0/1/2) | `severity` (Low/Warning/High) |
| `closed` (0/1) | `status` (Firing/Resolved) |
| `date` | `lastReceived` |
| `owner.name` | `assignee` |
| `department.name` | `department` |
