## Overview

The MongoDB Atlas provider enables Keep to receive alert notifications from [MongoDB Atlas](https://www.mongodb.com/atlas) monitoring. Atlas sends a JSON webhook POST to Keep whenever a monitoring alert threshold is breached or an alert is resolved.

This is a zero-configuration, webhook-only integration — no credentials or API polling required.

## How it works

```
MongoDB Atlas monitoring
        ↓  threshold exceeded / resolved
  HTTP POST JSON payload
        ↓
  Keep webhook endpoint
```

## Expected JSON payload

Keep parses the following fields from the Atlas webhook body:

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique alert ID |
| `created` | string | ISO-8601 timestamp when the alert was created |
| `updated` | string | ISO-8601 timestamp of the last update |
| `status` | string | `OPEN`, `CLOSED`, `ACKNOWLEDGED`, `TRACKING`, or `CANCELLED` |
| `acknowledgingUsername` | string | User who acknowledged the alert (if any) |
| `alertConfigId` | string | ID of the alert configuration that triggered |
| `clusterName` | string | Name of the Atlas cluster |
| `eventTypeName` | string | Atlas event category (e.g. `OUTSIDE_METRIC_THRESHOLD`) |
| `humanReadable` | string | Human-readable description of the alert |
| `metricName` | string | Metric name for metric-based alerts |
| `typeName` | string | Alert type: `HOST`, `REPLICA_SET`, `CLUSTER`, etc. |
| `currentValue` | object | `{ number, units }` for metric-based alerts |

Example payload (metric threshold exceeded):

```json
{
  "id": "5b8f8f8f8f8f8f8f8f8f8f81",
  "created": "2024-01-15T10:00:00Z",
  "updated": "2024-01-15T10:05:00Z",
  "status": "OPEN",
  "alertConfigId": "5b8f8f8f8f8f8f8f8f8f8f80",
  "clusterName": "prod-cluster-0",
  "eventTypeName": "OUTSIDE_METRIC_THRESHOLD",
  "humanReadable": "CPU usage on host atlas-abc123-shard-00-00 exceeded 80%.",
  "metricName": "NORMALIZED_SYSTEM_CPU_USER",
  "typeName": "HOST",
  "currentValue": { "number": 87.3, "units": "PERCENT" }
}
```

## Status → Keep alert status mapping

| Atlas status | Keep status |
|-------------|-------------|
| `OPEN` | FIRING |
| `TRACKING` | FIRING |
| `CLOSED` | RESOLVED |
| `CANCELLED` | RESOLVED |
| `ACKNOWLEDGED` | ACKNOWLEDGED |

## Severity mapping

Keep infers severity from the `typeName` and `eventTypeName` fields:

| Keywords | Severity |
|----------|----------|
| `NO_PRIMARY`, `REPLICATION_OPLOG_WINDOW_RUNNING_OUT`, `RS_STATE`, `MONGOS_IS_MISSING` | CRITICAL |
| `HOST_DOWN`, `DISK`, `OUTSIDE_METRIC_THRESHOLD`, `CONNECTION` | HIGH |
| `BACKUP`, `CREDIT_CARD`, `INSIDE_METRIC_THRESHOLD` | WARNING |
| All others | INFO |

## Connecting MongoDB Atlas to Keep

### 1. Add the MongoDB Atlas provider in Keep

In the Keep UI, go to **Providers → Add Provider → MongoDB Atlas**. No credentials are needed — save it to get your webhook URL:

```
https://api.keephq.dev/alerts/event/mongodbatlas
```

### 2. Configure the Atlas webhook

1. Log into [MongoDB Atlas](https://cloud.mongodb.com) and open your **Project**.
2. Navigate to **Project Settings → Integrations → Webhook Settings**.
3. Enable the webhook and paste your Keep webhook URL.
4. Add a custom HTTP header:
   - **Header name:** `X-API-KEY`
   - **Header value:** your Keep API key
5. Click **Save**.

### 3. Configure alert rules

In Atlas, go to **Alerts → Alert Settings** and create or edit alert configurations. Each alert configuration can send notifications to your webhook integration when its conditions are met.

### 4. Test the integration

Use **Alerts → Send Test Alert** in Atlas to send a test payload to your webhook URL. The alert should appear in Keep within seconds.
