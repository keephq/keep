## Overview

[Grafana Mimir](https://grafana.com/oss/mimir/) is an open-source, horizontally scalable, highly available, and multi-tenant time series database for Prometheus metrics. It is a drop-in replacement for Prometheus with additional capabilities including:

- **Multi-tenancy** via `X-Scope-OrgID` HTTP header
- Horizontal scaling across multiple nodes
- Long-term metric storage
- Compatible with existing Prometheus tooling and alerting rules

The Keep Mimir provider supports:
- **Pull mode**: fetches active firing alerts from Mimir's Prometheus-compatible API
- **Push mode**: receives alerts from Mimir's Alertmanager webhook receiver

## Connecting with the Provider

### Configuration

| Field | Required | Description |
|-------|----------|-------------|
| `url` | Yes | Mimir server base URL |
| `org_id` | No | Tenant/Org ID sent as `X-Scope-OrgID` header. Use `anonymous` for single-tenant. |
| `username` | No | Basic auth username (required for Grafana Cloud) |
| `password` | No | Basic auth password or Grafana Cloud API key |
| `verify` | No | Verify SSL certificates (default: `true`) |

### Grafana Cloud Mimir

```
URL:      https://prometheus-blocks-prod-us-central1.grafana.net/api/prom
Username: <your Grafana Cloud stack ID>
Password: <your Grafana Cloud API key>
Org ID:   (leave empty — not required for Grafana Cloud)
```

### Self-hosted Mimir (single-tenant)

```
URL:    https://mimir.example.com
Org ID: anonymous
```

### Self-hosted Mimir (multi-tenant)

```
URL:    https://mimir.example.com
Org ID: team-platform
```

## Webhook (Push) Configuration

Configure Mimir's Alertmanager to forward alerts to Keep using the Prometheus-compatible webhook receiver.

Upload an Alertmanager configuration to Mimir:

```bash
curl -X POST https://mimir.example.com/api/v1/alerts \
  -H "X-Scope-OrgID: <your-org-id>" \
  -H "Content-Type: application/yaml" \
  --data-binary @alertmanager.yaml
```

**alertmanager.yaml:**

```yaml
route:
  receiver: "keep"
  group_by: ['alertname']
  group_wait:      15s
  group_interval:  15s
  repeat_interval: 1m

receivers:
- name: "keep"
  webhook_configs:
  - url: "https://api.keephq.dev/alerts/event/mimir"
    send_resolved: true
    http_config:
      basic_auth:
        username: api_key
        password: "<your-keep-api-key>"
```

## Multi-Tenancy

Mimir requires the `X-Scope-OrgID` header for all API requests in multi-tenant deployments. Configure the `org_id` field in Keep to automatically include this header.

To monitor multiple tenants, create separate Keep provider instances — one per tenant:

- Provider `mimir-platform` → `org_id: team-platform`
- Provider `mimir-database` → `org_id: team-database`
- Provider `mimir-app` → `org_id: team-app`

## Alert Payload

Mimir Alertmanager webhook payloads follow the Prometheus Alertmanager format:

```json
{
  "receiver": "keep",
  "status": "firing",
  "alerts": [
    {
      "status": "firing",
      "labels": {
        "alertname": "HighCPUUsage",
        "severity": "critical",
        "instance": "web-01:9100",
        "__tenant_id__": "team-platform"
      },
      "annotations": {
        "summary": "CPU usage above 90%",
        "description": "CPU on web-01 has been above 90% for 5 minutes."
      },
      "startsAt": "2024-01-15T10:30:00Z",
      "endsAt": "0001-01-01T00:00:00Z",
      "fingerprint": "a1b2c3d4e5f60001"
    }
  ]
}
```

## Severity Mapping

| Mimir/Prometheus severity label | Keep severity |
|---------------------------------|---------------|
| `critical` | Critical |
| `error` / `high` | High |
| `warning` / `medium` | Warning |
| `info` | Info |
| `low` | Low |
| _(unset)_ | Info |
